# Netshoot KPI Toolkit Guide

This guide explains how the 3 netshoot files work together in your ContainerLab and how netshoot interacts with FRR routers.

## Files and Roles

### 1) netshoot.conf
Path: [configs/netshoot/netshoot.conf](configs/netshoot/netshoot.conf)

This is the shared runtime configuration file. Both scripts source it at startup.

What it defines:
- Lab/container naming: `LAB_PREFIX`, node container names (`PE2_LEFT`, `CE1`...)
- Endpoints for KPI traffic: `CE1_IP` ... `CE8_IP`
- QoS shaping parameters: `QOS_INTERFACE`, HTB classes, DSCP values
- Benchmark timing: ping/iperf durations, convergence timeouts
- Results location: `RESULTS_DIR=/results`

Think of this file as the single source of truth for addresses, interfaces, and KPI test tunables.

### 2) qos.sh
Path: [configs/netshoot/qos.sh](configs/netshoot/qos.sh)

Purpose:
- Apply, show, verify, reset QoS policy (HTB + SFQ + DSCP filters)

How it works:
- It runs `tc` and `ip` commands inside `pe2-left` namespace using Docker socket access.
- QoS is attached to `QOS_INTERFACE` (currently `eth1`) on `pe2-left`.
- Classes:
  - CUSTOMER-A (EF / `0xb8`) -> higher priority
  - CUSTOMER-B (AF21 / `0x48`) -> lower priority
  - Default class for unclassified traffic

Main commands:
- `apply`: install HTB tree and DSCP filters
- `show`: display qdisc/class/filter state with counters
- `verify`: return success only if all required objects exist
- `reset`: remove root qdisc

### 3) benchmark.sh
Path: [configs/netshoot/benchmark.sh](configs/netshoot/benchmark.sh)

Purpose:
- Run KPI scenarios and produce timestamped results

Output artifacts under `/results/<timestamp>/`:
- `raw.csv`
- `report.md`
- Convergence ping logs (`s4_ping.txt`, `s5_ping.txt`, `s6_ping.txt`)
- QoS JSON (`s7_cust_a.json`, `s7_cust_b.json`)

Scenarios:
- `srv6` (S0): router-native SRv6 control-plane validation from FRR/vtysh
- `srv6-impact` (S8): A/B service-impact test proving SRv6 VPN SID importance
- `latency` (S1-S3): ping + UDP jitter/loss + TCP throughput
- `convergence` (S4-S6): inject failures and measure recovery
- `qos` (S7): dual-stream contention + ratio check
- `all`: run everything

S0 exists specifically to prove SRv6 value comes from router configuration/state (not only external traffic generators). It records:
- IS-IS neighbors up on SRv6 core
- SRv6 locator and End.X advertisements in IS-IS DB
- Local PE SID inventory (End, End.X, End.DT4)
- seg6local routes in IPv6 RIB

S8 exists to prove operational importance, not only state visibility:
- Baseline reachability CE7 -> CE5
- Temporarily disable SRv6 VPN SID export on `pe2-right` (CUSTOMER-A)
- Measure degraded reachability
- Restore SID export and verify recovery

This provides direct before/after evidence that SRv6 VPN SID configuration is service-critical.

## How Netshoot Interacts with Routers

## Interaction Model

Netshoot does not run routing protocols directly. It acts as a test orchestrator.

It interacts with routers/CEs in two ways:

1. Namespace traffic generation
- Netshoot uses Docker socket and launches helper containers with:
  - `--net container:<target-node>`
- This makes ping/iperf run as if executed from inside that node namespace.
- Example: traffic from CE7 is generated by running a netshoot helper in CE7 netns.

2. Control-plane failure injection
- For convergence tests, script calls:
  - `docker exec <router> ip link set ... down/up`
  - `docker exec <router> vtysh ...` (for BGP shutdown/restore)
- This manipulates live FRR nodes to test recovery KPIs.

## Why Docker Socket Is Required

In [mpls_srv6.clab.yml](mpls_srv6.clab.yml), netshoot mounts:
- `/var/run/docker.sock:/var/run/docker.sock`

This allows netshoot scripts to create helper containers dynamically and target router namespaces.

## Quick Troubleshooting Tests

Run these from host (`/home/mohamed-khiari/frr`).

### Test 1: Script syntax and dependencies
```bash
bash -n configs/netshoot/qos.sh
bash -n configs/netshoot/benchmark.sh
docker exec clab-dual-plane-backbone-netshoot sh -lc '
  for x in docker python3 awk grep bc iperf3 ping tc; do
    echo -n "$x: "; command -v "$x" || echo MISSING;
  done'
```
Expected:
- No syntax errors
- All tools resolved inside netshoot

### Test 2: QoS lifecycle sanity
```bash
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/qos.sh reset
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/qos.sh apply
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/qos.sh verify
```
Expected:
- `verify` prints all `[OK]`

If failed:
- Check interface name in [configs/netshoot/netshoot.conf](configs/netshoot/netshoot.conf) (`QOS_INTERFACE`)
- Confirm `pe2-left` container is up

### Test 3: Latency KPI smoke test
```bash
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/benchmark.sh latency
```
Expected:
- CSV + report generated under `/results/<timestamp>/`
- `latency_*` and `ping_loss_pct` populated for S1-S3

### Test 3b: SRv6 router-native validation (important)
```bash
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/benchmark.sh srv6
```
Expected:
- `S0_srv6_native` metrics in CSV
- Non-zero `isis_db_srv6_locators`, `isis_db_srv6_endx`
- `pe2_left_locator_up = 1`
- Non-zero `pe2_left_sid_enddt4`

Troubleshooting files generated by S0:
- `s0_p1_isis_neighbor.txt`
- `s0_p1_isis_db_detail.txt`
- `s0_pe2_left_locator.txt`
- `s0_pe2_left_sid.txt`
- `s0_pe2_left_ipv6_route.txt`

### Test 3c: SRv6 service impact proof (very important)
```bash
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/benchmark.sh srv6-impact
```
Expected:
- `baseline_loss_pct` near 0
- `degraded_loss_pct` increases significantly (often 100)
- `recovery_loss_pct` returns near 0

Troubleshooting files generated by S8:
- `s8_baseline_ping.txt`
- `s8_degraded_ping.txt`
- `s8_recovery_ping.txt`

### Test 4: Convergence KPI smoke test
```bash
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/benchmark.sh convergence
```
Expected:
- `s4_ping.txt`, `s5_ping.txt`, `s6_ping.txt` created
- `pings_lost` present in CSV

Note:
- In highly redundant topology, measured outage may be `0 ms`.

### Test 5: QoS KPI smoke test
```bash
docker exec clab-dual-plane-backbone-netshoot /etc/netshoot/benchmark.sh qos
```
Expected:
- `s7_cust_a.json` and `s7_cust_b.json` created
- Throughput values reported in Mbps
- Ratio printed

If one stream is `N/A`:
- Open QoS JSON files and check `.error`
- Re-run once (script includes retry for transient control-socket failures)

### Test 6: Read latest results quickly
```bash
docker exec clab-dual-plane-backbone-netshoot sh -lc '
  L=$(ls -1dt /results/* | head -1);
  echo "Latest: $L";
  cat "$L/raw.csv"'
```

## Common Failure Patterns

- `docker: command not found` inside netshoot:
  - netshoot image missing docker CLI
- QoS verify fails:
  - wrong interface or tc not applied in target namespace
- QoS ratio near 1.0:
  - contention path may not hit the same shaped bottleneck
- Throughput `N/A`:
  - iperf control socket error in JSON

## Suggested Operational Flow

1. `qos.sh verify`
2. `benchmark.sh latency`
3. `benchmark.sh qos`
4. `benchmark.sh convergence`
5. Archive latest `/results/<timestamp>/`
